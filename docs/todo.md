# 今後の改善タスク (TODO)

精査によって特定された、`seadmin` の堅牢性とポータビリティを向上させるための改善項目です。

## 1. [完了] 外部コマンド実行時のロケール固定 (LC_ALL=C)
`sudo` や `ausearch`、`audit2allow` の出力を文字列マッチングで判定している箇所（"Permission denied" や "incorrect password" など）があります。
- **問題:** システムの `LANG` 環境変数によって出力が変化するため、特定の言語環境で判定に失敗する可能性がある。
- **対策:** `Command::new` で外部コマンドを呼び出す際、`env("LC_ALL", "C")` を設定して英語出力を強制する。
- **対応内容:** `src/selinux/commands.rs` と `src/sudo/runner.rs` の全ての `Command::new` に `.env("LC_ALL", "C")` を追加し、パース処理を英語のみに簡略化しました。

## 2. [完了] 安全な一時ファイル・ディレクトリの利用
`audit2allow_generate` で `/tmp` 直下にモジュール名に基づいた固定ファイルを作成していました。
- **問題:** マルチユーザー環境でのファイル名競合や、シンボリックリンク攻撃などのセキュリティリスクがある。
- **対策:** `tempfile` クレートなどを使用して、推測困難な一時ディレクトリを作成し、その中で作業を行う。
- **対応内容:** `tempfile` クレートを導入し、`audit2allow_generate` を修正。作業を一時ディレクトリ（`tempdir`）内で行い、結果の `.pp` ファイルは `/tmp` にランダムな名前（`seadmin-*.pp`）で生成するように変更しました。


## 3. [完了] XDG Base Directory への準拠
ログファイルの出力先が `~/.local/share/seadmin/seadmin.log` にハードコードされていました。
- **問題:** ユーザー設定（`XDG_DATA_HOME`）を尊重しておらず、標準的な Linux アプリケーションの挙動から外れている。
- **対策:** `dirs` クレートなどを利用し、環境変数に基づいた適切なディレクトリを選択するように修正する。
- **対応内容:** `dirs` クレートを導入し、`src/main.rs` 内のログパス解決を `dirs::data_dir()` を使用するように修正しました。これにより、標準の XDG データディレクトリ（通常は `~/.local/share`）が優先して使用されます。

## 4. [完了] `check_deps` における PATH 確認の改善
`semanage` や `audit2allow` などの管理コマンドは `/usr/sbin` や `/sbin` に配置されることが多いですが、一般ユーザーの `PATH` に含まれていない場合があります。
- **問題:** 実際にはインストールされていても、起動時のチェックで「不足」と判定される可能性がある。
- **対策:** 標準的な管理コマンドのパス（`/usr/sbin`, `/sbin`, `/usr/local/sbin`）も明示的にチェック対象に含める。
- **対応内容:** `is_in_path` 関数を修正し、`PATH` 環境変数に加えて、標準的な管理者用パスも直接チェックするようにしました。

## 5. [完了] `audit2why` 解析ロジックの堅牢化
`audit2allow -w` の人間向けテキスト出力をパースして原因を特定しています。
- **問題:** `policycoreutils` のバージョンアップによって出力形式が変更された際、解析が壊れる可能性がある。
- **対策:** 可能な限り機械読み取りに適した形式（もしあれば）を利用するか、より柔軟な正規表現等によるパースを検討する。
- **対応内容:** 特定のヘッダー（"Was caused by:"）に依存せず、出力全体から重要なキーワード（`missing type enforcement`, `allow rule`, `target context label` 等）を小文字変換して柔軟にスキャンするように変更しました。

## 6. [完了] テスト網羅性の向上とコマンド生成の検証
現状、解決策（Remedy）から実行される具体的なコマンド（`semanage`, `setsebool` 等）の生成ロジックにテストがなく、多様な拒否パターンへの対応が未検証でした。
- **課題:** ファイル、ポート、Boolean、カスタムポリシーなど、各パターンで正しいコマンド引数が生成されるか担保されていない。
- **対策:** 「データ駆動型テスト（Golden Master Test）」を導入。様々な AVC ログ（生テキスト）を入力とし、期待される Remedy と実行コマンドが生成されるかを検証する。
- **対応内容:**
    - `src/ui/screens/avc_detail.rs` に `build_options` のテストを実装し、生成コマンドの正しさを検証。
    - `src/selinux/avc.rs` に集約ロジック（重複カウント）やブロック間パス補完の高度なテストを追加。
    - `src/ui/app.rs` に解決済み状態の永続性（再ロード時の引き継ぎ）のテストを追加。
    - `diagnose_remedy` の優先順位を修正（Boolean 判定をポート判定より優先）し、より正確な診断を可能にしました。
